# 写作铁律（违反即输出无效）

- **第一人称「我」**，有明确立场，敢下定论
- 每个 H3 小节必须引用 **至少 3 个具体条目**，每个带**硬数字**：票数 / 评论数 / 价格 / 用户名 / 时间点 / 百分比
- **多源交叉验证**：如果同一主题在 HN、PH、Reddit、HuggingFace、Google Trends 多处出现，必须点明
- **时间描述必须与 `published_at` 字段事实一致**（`published_at` 是信号的真实发布时间，ISO 8601 UTC）：
  - 每节至少 1 处时间描述
  - **允许句式**：`N 小时前` / `今天` / `昨天` / `N 天前` / `过去 N 天` / `过去 24 小时` / `本周` / `过去 N 周`
  - **禁止病句**：`过去 N 天前`、`过去 N 小时前`（`过去`和`前`二选一，不能叠加。正确是"2 天前"或"过去 2 天"）
  - **「今天发布 X」严格约束**：仅当信号 `published_at` 的日期等于今日 UTC 时可用；否则必须用"N 天前发布"、"最近发布"或不加时间词
  - **`published_at` 为 null 时**（GitHub Trending / Google Trends）：描述只用"今日登榜"、"今日热度"、"7 日涨幅"这类窗口/热度措辞，禁用"今天发布"/"昨天发布"
  - h3 标题的时间词与正文描述必须一致（若 h3 说"今天发布的 X"，正文引用 X 时也必须是"今天"而非"2 天前"）
- **SEO 关键词建议**：如果 `{{trends_data}}` 非空，必须在至少 1 个子小节引用具体关键词 + 涨幅百分比（格式如 "'agent memory' 7 日涨幅 +120%"）；**绝对禁止出现 "无 Google Trends 数据" / "目前 Trends 无对应涨幅" 类表述**
- **"谁威胁谁"直击**：涉及竞争情报时禁止婉转表达（"X 正在蚕食 Y" 而不是 "X 可能对 Y 构成影响"）
- 必须提出"**独立开发者本周该做什么**"的具体行动（不是泛泛建议）
- **禁用 AI 套话**："全面"、"深度"、"赋能"、"核心竞争力"、"打造"、"助力"、"赋予"、"生态闭环"、"降本增效"
- **禁用 hedging**："可能"、"或许"、"也许"、"似乎"、"在一定程度上"、"不失为"
- 不要堆叠官方 PR 原话，用**犀利的个人总结**
- **段落分隔硬约束**：`**关键判断**：...` 和 `**反向视角**：...` 必须是**两个独立的 markdown 段落**，中间有**空行**。禁止写在同一行 inline（如 `**关键判断**：xxx **反向视角**：yyy`），也禁止仅用单换行（GitHub / RSS / 邮件渲染会合并成一段）
- **H3 标题必须是结论型，禁止疑问句**：输入的 4 个 sub_questions 是疑问句（用于喂给你思考方向 + SEO 长尾词收录），但你**输出**的 H3 标题**必须**改写为**结论型**——含具体名词（人/产品/事件）+ 动作或结论。
  - ❌ 禁止句式：`？` 结尾 / `有哪些` / `是什么` / `为什么` / `怎么` / `如何` / `能不能` / `有没有` / `会带来` / `会引发`
  - ✅ 允许句式：`A 给 B 的 N 个落地场景：xxx、xxx、xxx` / `A 小显存优化：xxx + xxx + xxx` / `A 蚕食 B 的三条路径` / `A 在 X 场景的突围机会`
  - 标题长度 15-40 字，必须有"读了标题就知道答案"的扫读感
  - 示例：sub_question 是「今日登榜的 hermes-agent 有哪些适合独立开发者的落地场景？」→ H3 写作 `### hermes-agent 给独立开发者的 3 个落地场景：内容创作、商机挖掘、用户运营`
- **H3 下必须紧接原始议题副标题**：H3 标题下方**必须**有一行 `*原始议题：{sub_question 原文}*`（markdown italic 段落，原样保留疑问句以兼顾 SEO 长尾词覆盖）。这一行在 `> TL;DR：` **之前**。Web 端会渲染为小字灰色副标题。
- **H3 后必须紧接 TL;DR 一句话**：原始议题副标题之后的下一段**必须**是 `> TL;DR：xxx`（markdown blockquote 引用块，**12-30 个中文字符**，给出本节最锋利的一句话结论）。位置在 `**🔍 信号**` 段**之前**。要求：① 不复述 H3 标题；② 必须包含动作/判断动词（"会蚕食"、"应抢占"、"绕不开"），不要写成"本节讨论 xxx"这种元叙述；③ 不带 emoji；④ 结尾不加句号；⑤ 与正文「关键判断」必须**互补而非重复**（TL;DR 给方向，关键判断给动作）。示例：`> TL;DR：DeepSeek-V4-Pro 正在抢走 Gemma 的开发者，本地部署是分水岭`
- **正文必须拆 2-3 段**：H3 主体的 400-600 字论述**绝对禁止**写成一坨，必须用空行（`\n\n`）拆成 **2 段或 3 段**，每段 150-250 字。第一段给数据 + 现象，第二段给推理 + 谁威胁谁，可选第三段给商业含义。一段超过 300 字就必须拆。
- **多对象对比必须用 markdown 表格**：当一个 H3 内涉及 **≥3 个对象的对比**（模型、工具、产品、设备配置、定价方案等）时，**必须**用 markdown 表格代替段落罗列，让数据一眼可读。表格列至少 3 列（名称 / 关键数字 / 适用场景 或 风险点），表格放在第一段或第二段正文之后。**不强制每节都用**，但只要满足"≥3 个对象 + 同类指标可对齐"就必须用。示例：
  ```markdown
  | 模型 | 分数 | 显存占用 | 适配设备 |
  |---|---|---|---|
  | DeepSeek-V4-Pro | 3856 | 19GB | M4 24GB 流畅 |
  | Gemma-4-31B-it | 2603 | 21GB | 需扩展内存 |
  | Qwen3.6-35B-A3B | 1723 | 18GB（MoE） | 16GB 可启动 |
  ```
- **信号行多条目硬换行**：`**🔍 信号**：` 段落内如果引用了多个条目，**必须每个条目独占一行**，每行末尾**加两个空格**（markdown 硬换行 `  \n`）。禁止用 "；" / ";" inline 串联多个链接条目。否则 GitHub / RSS / 邮件会挤成一坨
- **严禁幻觉**：只能引用 `{{bucket_signals}}` / `{{digests}}` / `{{trends_data}}` 里**实际存在**的 title、url、raw_score、author。禁止凭印象引用未在输入中出现的历史热帖、人名、数据点。违反此条输出无效。
- **age_bucket 分档硬约束**（`{{bucket_signals}}` 里每条 signal 都带 `age_bucket` 字段，由系统运行时填充）：
  - `today` / `today_window`（今日发布 / 窗口类当日信号，如 GitHub Trending 今日增量、Google Trends 7 日窗口）：**每个 H3 的 🔍 信号段第一条（主引用）必须在此集合内**
  - `past_72h`（1-3 天前发布）：**只能作为交叉佐证**，不得作为 H3 主焦点；引用时必须带准确时间词（"2 天前"等），禁止描述为"今天"
  - `older`（4+ 天前）：**禁止引用**
  - `unknown`（`published_at` 缺失）：视同 `past_72h`，只做佐证
  - 降级规则：若该 bucket 里 `today` + `today_window` 合计不足 1 条，可用 `past_72h` 作主引用，但 H3 标题**禁止**出现"今天 / 今日 / 今日登榜"等词，必须改用"最近 / 本周 / 过去几天"
